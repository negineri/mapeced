# MAP-Eの非連続ポートレンジ算出アルゴリズム

この文書は、`docs/ip6tnl0.nft` と `docs/tc-ip6tnl0.sh` で使っている  
「MAP-Eで許可される非連続ポート集合」を導出する考え方をまとめたものです。

## 背景

MAP-Eでは CE ごとに使えるポートが **連続範囲ではなく、ビット条件で決まる離散集合** になります。  
一方、`nftables` の `snat to ...:start-end` は連続1レンジしか扱えません。

この差を埋めるために次の2段構えを使います。

1. `nftables` で「同数の連続ポート」を一旦割り当てる
2. `tc pedit` でビットを全単射（1対1対応）変換し、MAP-Eが要求する非連続集合へ写像する

## ポートのビット表現

16bitポート番号を、下記の3領域に分けます。

- `A`: オフセットビット数（a bits）
- `PSID_LEN`: PSIDのビット長
- `M`: 下位連続ポートビット数（j bits）

`A + PSID_LEN + M = 16`

通常はAとPSID_LENが与えられ、Mは計算します。

ポート `p` は以下で表せます。

`p = (a << (PSID_LEN + M)) | (psid << M) | j`

- `a` は `a_min .. (2^A - 1)`
- `j` は `0 .. (2^M - 1)`

よって、MAP-Eで利用可能なポート集合は次の和集合です。

`S = ⋃_{a=a_min..2^A-1} [ (a << (PSID_LEN + M)) + (psid << M), (a << (PSID_LEN + M)) + (psid << M) + (2^M - 1) ]`

各ブロック幅は `2^M`、ブロック間隔は `2^(PSID_LEN+M)` なので、見た目は非連続になります。

`a_min` は固定値1ではなく、MAP-Eで除外したい先頭ポート帯（例: Well-known）から決まります。  
除外上限を `P_exclude_max`（inclusive）とすると、実装上は次で求めます。

`a_min = max(1, ceil((P_exclude_max + 1) / 2^(PSID_LEN + M)))`

（同値変形: `ceil((P_exclude_max + 1) / (2^PSID_LEN * 2^M))`）

## 具体例（`A=4`, `PSID_LEN=8`, `PSID=0xab`）

このとき（`P_exclude_max=1023` とすると）利用可能ブロックは `a=1..15` の15個、各16ポートです。

- `0x1ab0-0x1abf`
- `0x2ab0-0x2abf`
- ...
- `0xfab0-0xfabf`

```
[  a  ][    PSID    ][  j  ]
<- A -><- PSID_LEN -><- M ->
```

Mは`M = 16 - A - PSID_LEN`で表せます。

合計ポート数は:

`(2^A - a_min) * 2^M`

上の具体例では `a_min=1` なので `15 * 16 = 240` です。

## 連続レンジを経由する理由

`nftables` では非連続集合 `S` を直接 `snat` 指定しづらいため、同じ個数の連続集合 `C` を使います。

- `PORT_START = (1 << 15) + (a_min << M)`
- `PORT_END   = PORT_START + (2^A - a_min) * (2^M) - 1`

最上位ビットを立てている理由はwell-knownポートを避けるためです。

これにより`C`は

```
[  1  ][      0pad      ][  a  ][  j  ]
<- 1 -><- PSID_LEN - 1 -><- A -><- M ->
```

`A=4, M=4, a_min=1` なら `32784-33023`（`0x8010-0x80ff`）です。

## 全単射写像（tcでのビット変換）

連続集合 `C` のポートを次の規則で変換します。tcで処理するために対象となるパケットには事前に `fmark` を付与しておきます。

1. `C` は全てのポートで最上位ビット `0x8000` が立っているため、 `a` の最上位ビットが立っていない場合は `0x8000` のビットを下ろす。
2. 残りの `a` のビットを確認し、立っている場合は対応する上位ビットを立てる
3. `psid` を `bits[M+PSID_LEN-1 : M]` に埋め込む
4. TCP/UDPはchecksumを再計算

これは単なるビット置換＋固定値埋め込みなので、`C` と `S` の間で 1対1 対応を作れます。  
そのため、SNATの割り当てを壊さずに、MAP-E要件のポート集合へ正確に変換できます。

## 逆方向（戻りパケット）

受信側では逆変換を行います。

1. `PSID` 部分でフィルタし、対象フローへ `fwmark` を付与
2. `PSID` 部分を0で埋める（`a` が移動してくる）
3. `a` のビットを全て確認し、対応する下位ビットを立てる
4. 最上位ビットを立てる
5. TCP/UDPはchecksumを再計算

これらをTCP/UDPとICMP内包ヘッダに適用しています。

## 実装時の注意

- `A = 0`は連続ポートの表現としてRFCで定義されている。この場合はtcによる変換処理は不要
- `A + PSID_LEN < 16` を満たすこと（`M > 0` が必要）
- `PSID` は `PSID_LEN` ビットに収まる値を使うこと
- `a_min` を環境ごとの除外方針（例: Well-known除外）から計算すること
- ICMPエラーでは「内包される元パケットヘッダ」のオフセット（例: 48byte）を正しく扱うこと

---

上記を使うと、`nftables` の単一連続レンジ制約を維持したまま、MAP-Eの非連続ポート条件を満たせます。
